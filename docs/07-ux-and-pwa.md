# 07. UX と PWA

モバイルファーストで設計し、PWA として提供する。

---

## 7.1. 設計原則

### 7.1.1. 「2 タップで問題開始」

- 通知 / ホーム画面アイコン → ホーム画面 → Drill カードタップ → 即出題
- ログイン画面を挟まない (Passkey 認証後の HTTP-only cookie で自動継続、ADR-0004)
- スプラッシュ画面は最小限

### 7.1.2. 「片手操作」

- ボタンは画面下部に集約 (親指リーチ圏内)
- スクロール不要で 1 問が収まるレイアウト
- 選択肢ボタンは大きめ (48px 以上)

### 7.1.3. 「10 秒でスキップできる任意項目」

- 「なぜそう答えたか」入力は任意、10 秒でスキップ可能
- 自己評価は 1-5 の絵文字ボタン (タップ 1 回)

### 7.1.4. 「読むより解く」

- 解説は折りたたみで表示、ワンタップで次へ進める
- 長文は避ける、図やコードで補う

## 7.5. PWA 対応

### 7.5.1. 必要要素

| 要素                         | MVP (#24)      | 内容                                                           |
| ---------------------------- | -------------- | -------------------------------------------------------------- |
| `manifest.webmanifest`       | ✓              | アイコン、名前、`display: standalone`、テーマ色                |
| Service Worker (static 限定) | ✓              | `/_next/static/` / icons / manifest / favicon のみ Cache-First |
| App Shell (HTML) offline     | Phase 5+ (#40) | auth-aware な HTML cache 設計が必要なため                      |
| API offline fallback         | Phase 5+ (#40) | endpoint allowlist + logout 時 purge が必要                    |
| Web Push                     | Phase 5+ (#37) | iOS 実運用の検証後に導入                                       |

MVP (issue #24) では「PWA としてインストール可能」「静的資産を cache-first で高速化」
までを満たす。オフライン時の HTML/API fallback は credentialed レスポンスの残留リスク
(ログアウト後のデータ復活、アカウント切替時の混在) を避けるため、
auth-aware な設計 (issue #40) までスコープ外とする。

### 7.5.2. Manifest

真実の源は `public/manifest.webmanifest`。以下は抜粋 (docs として維持)。
アイコンはリポ直下の `public/icon-192.png` `public/icon-512.png` と
Apple 用 `public/apple-icon.png` (180x180)。`maskable` と通常表示を
1 つのアイコンで兼用 (`purpose: "any maskable"`)。

```json
{
  "name": "Tanren",
  "short_name": "Tanren",
  "description": "エンジニアのための AI 家庭教師",
  "start_url": "/",
  "scope": "/",
  "display": "standalone",
  "orientation": "portrait",
  "background_color": "#ffffff",
  "theme_color": "#0f172a",
  "lang": "ja",
  "icons": [
    { "src": "/icon-192.png", "sizes": "192x192", "type": "image/png", "purpose": "any maskable" },
    { "src": "/icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any maskable" }
  ]
}
```

### 7.5.3. Service Worker の戦略

**MVP (issue #24)**: プライバシー優先で cache 範囲を最小化。
`public/sw.js` で以下のみ:

- **/\_next/static/\*, /icon-\*, /manifest.webmanifest, /favicon.ico**: Cache-First
  (Content-Hash 付き静的資産、stale 化しない)
- **/api/\* + ドキュメント navigation (HTML) + /login**: SW 介入せずネットワーク直通
  (credentialed レスポンスを cache storage に残さない)。ログアウト後やアカウント切替時に
  古い個人データが復活するリスクを回避。

**Phase 5+ (issue #40)**:

- **問題の先読み**: Stale While Revalidate (数問先を裏で取得)
- **解答の送信**: Background Sync (電波復帰時に自動送信)
- **Insights Dashboard のオフラインキャッシュ** (endpoint allowlist 設計要)

### 7.5.4. オフライン対応の範囲

**MVP**:

- **できる**: ホーム画面アイコンから起動 (manifest)、PWA としてインストール、
  オフライン時にブラウザキャッシュ (HTTP cache) 由来で直前のページを開ける程度
- **できない**:
  - 以前解いた問題の履歴閲覧 (HTML / API ともに SW 非介入)
  - 先読みされた問題を解く (Phase 5+ 予定)
  - Insights Dashboard のオフライン表示 (Phase 5+ 予定)

Phase 5+ (issue #40) で上記の「できない」リスト (HTML / API の offline 対応、
先読み問題の offline 実行、Insights のオフライン表示) を実装する。現 MVP は
「PWA としてインストール可能」「起動時の静的資産を cache-first で高速化」までが狙い。

### 7.5.5. Web Push (Phase 5+ 予定)

MVP には入れない。理由:

- iOS 16.4+ の PWA は **ホーム画面追加した場合のみ** Web Push を購読できる
- iOS は購読が不安定 (バックグラウンド動作制約)
- サーバー側の VAPID 鍵管理・cron 一斉配信のコストに見合わない
- **MVP では代替として Resend によるメール配信** (`7.10` 参照)

Phase 5+ で実運用テスト (iOS で確実に届くか) をした上で導入を判断。

## 7.10. 通知戦略

### 7.10.1. 配信手段: メール優先

Web Push は iOS 制約が大きいため、**MVP では Resend によるメール**を主通知路とする。
Web Push は Phase 5+ で実検証後に導入。

### 7.10.2. 通知の種類と設計

| 種類                | タイミング | 手段           | 本文例                             | 追加 Phase |
| ------------------- | ---------- | -------------- | ---------------------------------- | ---------- |
| Daily reminder      | 設定時刻   | メール         | 「今日の復習 12 問が待ってます」   | Phase 3    |
| Weekly digest       | 日曜 9:00  | メール         | 「先週の学びレポートが届きました」 | Phase 3    |
| Blind spot unlocked | 随時       | アプリ内バナー | 「Paxos が解禁されました」         | Phase 3    |

Streak at risk は **やらない** (Streak 自体を出さない方針)。

### 7.10.3. 通知を嫌われない工夫

- すべての通知を設定画面で個別 on/off 可能
- デフォルトで過度に送らない (Daily + Weekly のみ)
- 本文は具体的 ("12 問" など数字を入れる)
- 2 週間開いていないユーザーには追加で送らない (義務感を生まない)

## 7.11. 初回オンボーディング

### フロー

```
[Passkey 登録 (初回のみ、`navigator.credentials.create()`)]
    ↓
[ウェルカム画面: 3 枚]
  1. 「AI がエンジニアのための問題を出します」
  2. 「忘れる前に再出題する科学的な仕組み」
  3. 「あなたの学習状態を診断します」
    ↓
[興味分野選択]
  ☑ programming  ☑ network  ☐ security ...
    ↓
[レベル自己申告]
  beginner / junior / mid / senior / staff
    ↓
[初期診断テスト (3-5分, 20問)]
  各分野から数問 → 初期 mastery 設定
    ↓

以降は Passkey 登録 → ログインの繰り返し。

## 7.12. エラー / エッジケース

### 7.12.1. LLM 失敗時

- タイムアウト → キャッシュから代替問題を返す
- 採点失敗 → 再試行 3 回 → ダメなら「後で確認します」扱い (後で自動再採点)

### 7.12.2. ネットワーク切断

- 解答は IndexedDB に保存 → 復帰時に送信
- 「オフライン中です」表示 + リトライボタン

### 7.12.3. 空データ

- 初心者に「今日の復習 0 問」表示 → 代わりに「新しい concept を試そう」誘導
- ドメイン未着手 → 「まずは診断テストから」誘導

## 7.13. 「詳しく聞く用にコピー」機能

### 7.13.1. 目的

採点後に「もう少し深掘りしたい」と思ったとき、**Tanren 側で追加対話を抱え込まず、
ユーザーの既存 ChatGPT / Claude セッションに橋渡しする**。

- Tanren 側の API コストをかけずに深掘りできる
- ユーザーが普段使う AI (好みのモデル・コンテキスト) で学習を継続できる
- 「話す」学習に Tanren 内で流れず、解くことへの集中を維持できる

### 7.13.2. 配置

- `/drill` 採点結果画面 (`7.2.3`)
- `/insights/history` の各 attempt 詳細

### 7.13.3. 挙動

1. ボタン押下 → 下記テンプレートに該当 attempt の情報を埋めて `navigator.clipboard.writeText`
2. トースト表示 "コピーしました。ChatGPT / Claude に貼って深掘りしてください"
3. (オプション) 副アイコンで **chatgpt.com / claude.ai を新タブで開く** ショートカット
4. コピー操作を `attempts.copied_for_external` (カウント) に記録して Insights でトラック

### 7.13.4. プロンプトテンプレート (日本語)

```

2. この概念の背景 (なぜこういう仕組みになっているか)
3. 関連する概念や、実務でのハマりどころ

### 7.13.5. 実装メモ

- テンプレートは `src/lib/share/copy-for-llm.ts` に 1 ファイルで
- 変数は安全に挿入 (改行エスケープ不要、ただし ` ``` ` バッククォートは置換で対処)
- 長すぎる回答は頭 2000 文字で切って `...` を付ける
- モバイル: Share Sheet (`navigator.share`) で「コピー / ChatGPT / Claude / その他」を出せるとベター

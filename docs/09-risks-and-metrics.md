## 9.2. リスク詳細と対策

### 9.2.1. R1: LLM 採点の誤判定

**問題:**
- 「正解なのに ❌ にされた」体験が続くと学習意欲を削ぐ
- 特に記述系 (written) で発生しやすい

**対策:**
- 各採点結果に「これは正解だ」反論ボタン
- 反論されたら別モデル (上位モデル) で再採点
- 結果に応じて `attempts.correct` を更新
- 反論ログを蓄積 → プロンプト改善の原資
- ルーブリックを明示することで、なぜその判定になったかを透明化

### 9.2.2. R2: 問題品質のバラつき

**問題:**
- LLM 生成の問題には「明らかに質が低い」ものが混ざる
- 曖昧な設問、複数正解、事実誤認など

**対策:**
- 各問題に 👍 / 👎 ボタン (任意)
- 👎 が一定数 → 問題を非表示 + 分析
- 初期の 1-2 ヶ月は自分で生成結果をレビューし、プロンプト改善
- 事前生成問題を一度人力確認 (Phase 2)

### 9.2.3. R3: API コスト暴発

**問題:**
- 思ったよりユーザーが問題を解き、API コストが収益を食う
- 悪意あるユーザーがスパム的に問題生成

**対策:**
- キャッシュ戦略で 50%+ を既存問題から提供
- プロンプトキャッシングで入力 90% 割引
- Free プランは 1 日 10 問制限
- Pro プランも 1 分 10 問 (バースト防止)
- 月次の API 予算アラート設定
- 事前生成バッチで繁忙時の API 負荷を平準化

### 9.2.4. R4: 問題マンネリ

**問題:**
- 同じ concept で似た問題が繰り返し出る
- ユーザーが答えを暗記してしまう

**対策:**
- 問題生成プロンプトに「過去 30 日の出題要約」を含める
- 「これらとは違う角度で」と LLM に指示
- 思考スタイルをローテーション
- 同一 concept × 同一スタイルで連続しないよう制御
- Insights で「過去にどう問われたか」を可視化して学習者側も意識

### 9.2.5. R5: 継続率低下

**問題:**
- Anki すら続かない人が多い
- 学習アプリの平均 D30 retention は 20% 程度

**対策:**
- Daily reminder 通知 (適切な時刻に)
- Streak at risk 通知 (夜に「今日 0 問ですよ」)
- Weekly digest で学習の成果を可視化
- 「2 分で終わる最小ドリル」モード (気軽に開始)
- Insights で成長実感を強化
- 初期 1 週間が勝負 → オンボーディングで「最初の成功体験」を作る

### 9.2.6. R6: 知識ツリー不足

**問題:**
- 初期の concept が少ないとすぐ底が見える
- 偏ったシードだと特定ユーザーに刺さらない

**対策:**
- MVP 時点で 100 concept
- β 公開までに 200 concept
- 公開後 6 ヶ月で 500 concept
- ユーザーから「この concept が欲しい」要望を受付
- 動的 concept 追加機能 (Custom で未登録が出たら追加)

### 9.2.7. R9: データ漏洩

**問題:**
- 学習履歴は個人情報
- 誤答履歴は「弱点」なので公開されたくない

**対策:**
- Turso の at-rest 暗号化
- Clerk による認証、JWT ベース
- すべてのクエリに `user_id` フィルタ必須
- PostHog にログ送る際に PII を除去
- 退会時に完全削除 (GDPR / APPI 準拠)

### 9.2.8. R10: プロンプト変更による揺れ

**問題:**
- プロンプトを改善すると過去の採点基準と乖離
- 「前は ✅ だったのに今は ❌」が起きる

**対策:**
- 各 attempt に `prompt_version` 記録
- プロンプト変更は feature flag で段階適用
- 過去の判定は変えない (新プロンプトは新しい attempt から)
- A/B テストで効果検証

## 9.3. 成功指標 (メトリクス)

### 9.3.1. ノース・スター指標

**「週次 active_days / user」**
- 週に何日アプリを使ったか
- 学習習慣がついたかの最良の指標

### 9.3.2. 主要指標 (North Star の先行指標)

#### Engagement

- **D1 retention** (翌日復帰率)
- **D7 retention**
- **D30 retention**
- **Median session duration** (中央値の学習時間)
- **Median questions per session** (中央値の問題数)

#### Learning

- **Mastery concepts per user** (習得 concept 数)
- **Mastery growth rate** (週次の mastery 増加量)
- **Misconception resolution rate** (誤概念が矯正された率)
- **Accuracy trend** (正答率の時系列推移)

#### Product

- **Custom Session 利用率** (全セッションのうち Custom の割合)
- **Insights 画面滞在時間** (ユーザーが自己診断に使う時間)
- **Feedback button press rate** (👍👎 の押下率)
- **Reason-given rate** (誤答時の「なぜ」入力率)

#### Business

- **Free → Pro conversion rate**
- **Monthly Active Users (MAU)**
- **Monthly Recurring Revenue (MRR)**
- **Cost per active user** (API コスト / MAU)
- **Gross margin** (売上 - API 原価)

## 9.5. 測定方法

### 9.5.1. PostHog

- ユーザー行動の全イベント記録
- ファネル分析 (サインアップ → 初回 drill → D7 復帰)
- Feature flag による A/B テスト

### 9.5.2. 自前 DB

- 学習関連 (mastery、attempts) は PostHog ではなく DB で集計
- `daily_stats` テーブルに日次集計キャッシュ

### 9.5.3. アラート

- Sentry でエラー率監視
- Vercel でパフォーマンス監視
- Slack に日次サマリ通知 (BOT)

## 9.6. 失敗時の判断基準 (No-Go ライン)

### Phase 1 終了時 (Week 3)

- 自分が「続けたい」と思えない → 方向転換検討

### Phase 4 終了時 (Week 8, α 公開)

- α ユーザー 3 人中 1 人以下しか 1 週間使わない → 設計見直し

### Phase 5 終了時 (Week 12, β 公開)

- D7 retention 20% 未満 → 要因分析 + ピボット検討

### Phase 6 終了時 (Week 16, 有料化)

- 有料会員 5 人未満 → プライシング / 機能見直し

## 9.7. ピボット候補 (失敗時の方向転換)

もし現在の設計で成立しなかった場合:

### P1: 法人向けにピボット

- 企業の教育研修ツールとして提供
- 管理者ダッシュボード、チーム学習状況
- カスタム知識ツリー (社内用語対応)

### P2: 特定分野に特化

- 「セキュリティ資格対策」「面接対策」など範囲を絞る
- 既存の学習者の specific な needs に応える

### P3: 教える側のツールにピボット

- エンジニアリング教育者向けに問題生成ツール
- メンターが部下の学習状況を把握するツール

### P4: ChatGPT/Claude Plugin 化

- スタンドアロンアプリをやめ、Claude Projects / Custom GPT として提供

## 9.8. 撤退基準

以下が揃ったら撤退を検討:

- Week 24 時点で MAU 100 人未満
- 月次コスト > 月次収入
- 自分自身が続けていない
- ユーザーからのフィードバックがほぼない
- 代替サービスが急速に台頭

撤退するときは:
- β ユーザーへ丁寧に通知
- データエクスポート機能を提供
- 学びを次のプロジェクトに活かす

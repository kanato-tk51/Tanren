# 09. リスクと成功指標

個人開発プロジェクトとして、想定される失敗パターンと、成功を測るための指標。

---

## 9.1. リスク一覧

| # | リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|---|
| R1 | LLM 採点の誤判定で萎える | 高 | 中 | 反論ボタン → 再採点 |
| R2 | 問題の品質バラつき | 高 | 中 | 👍👎 フィードバック、プロンプト改善 |
| R3 | API コスト暴発 | 中 | 低 | キャッシュ + 予算アラート + レート制限 |

| R6 | 知識ツリーが薄い / 偏る | 中 | 中 | MVP 時点で 100 concept 確保 |
| R7 | Custom Session の NL パース失敗 | 中 | 中 | プレビュー + 手動編集 UI |

## 9.2. リスク詳細と対策

### 9.2.1. R1: LLM 採点の誤判定

**問題:**
- 「正解なのに ❌ にされた」体験が続くと学習意欲を削ぐ
- 特に記述系 (written) で発生しやすい

**対策:**
- 各採点結果に「これは正解だ」反論ボタン
- 反論されたら別モデル (上位モデル) で再採点
- 結果に応じて `attempts.correct` を更新
- 反論ログを蓄積 → プロンプト改善の原資
- ルーブリックを明示することで、なぜその判定になったかを透明化

### 9.2.2. R2: 問題品質のバラつき

**問題:**
- LLM 生成の問題には「明らかに質が低い」ものが混ざる
- 曖昧な設問、複数正解、事実誤認など

**対策:**
- 各問題に 👍 / 👎 ボタン
- 👎 が付いたら生成プロンプトを見直し
- 初期は自分で生成結果をレビューし、プロンプトを反復改善
- 事前生成問題を一度自分で確認してから本番投入 (Phase 2)

### 9.2.3. R3: API コスト暴発

**問題:**
- バグによる無限ループで API を連発してしまう
- 夢中で解きすぎて月額が跳ねる

**対策:**
- キャッシュ戦略で 50%+ を既存問題から提供
- プロンプトキャッシングで入力 90% 割引
- Anthropic 側の**月間予算アラート**を設定 (例: $20 超えたら通知)
- 1 分 10 リクエストの軽いスロットリング (バグ保険)
- 事前生成バッチで繁忙時の API 負荷を平準化

### 9.2.4. R4: 問題マンネリ

**問題:**
- 同じ concept で似た問題が繰り返し出る
- 答えを暗記してしまう

**対策:**
- 問題生成プロンプトに「過去 30 日の出題要約」を含める
- 「これらとは違う角度で」と LLM に指示
- 思考スタイルをローテーション
- 同一 concept × 同一スタイルで連続しないよう制御
- Insights で「過去にどう問われたか」を可視化

### 9.2.5. R5: 自分が継続しない

**問題:**
- 作ったのに 2-3 週間で開かなくなる
- Anki と同じ末路

**対策:**
- Daily reminder 通知 (適切な時刻に、Phase 3+ でメール配信)
- Weekly digest で成長を可視化して自己強化 (Phase 3+、メール)
- 「2 分で終わる最小ドリル」モード (気軽に開始)
- Insights で成長実感を強化
- **ホーム画面アイコンを目立つ位置に** (物理的な導線確保)
- **作者自身が楽しい UI/UX にこだわる** (自分が使うから)

Streak at risk 通知のような「義務感を生む」通知は意図的に作らない。
Streak 自体を可視化しない (`07.2.1` / `OPEN_QUESTIONS.md Q1` 参照)。

### 9.2.6. R6: 知識ツリー不足

**問題:**
- 初期の concept が少ないとすぐ底が見える
- 偏った seed だと飽きる

**対策:**
- MVP 時点で 100 concept 用意
- 業務で出てきた概念を随時追加する習慣を作る
- Custom Session で未登録が出たら動的追加
- 半年で 300-500 concept を目標にゆるく整備

### 9.2.7. R9: プロンプト変更による揺れ

**問題:**
- プロンプトを改善すると過去の採点基準と乖離
- 「前は ✅ だったのに今は ❌」が起きる

**対策:**
- 各 attempt に `prompt_version` 記録
- 過去の判定は変えない (新プロンプトは新しい attempt から)
- 体感で大きくズレた場合はプロンプトをロールバック

### 9.2.8. R12: 作るのに飽きて放置

**問題:**
- 個人開発あるあるの「作るのが目的化して使わなくなる」
- 逆に使うことばかりで改善しなくなる

**対策:**
- **作るフェーズと使うフェーズを分けない**、常にドッグフードしながら改善
- 1 週間に 1 回は小さな機能追加か改善を入れる
- 義務感で続けない、楽しくなければ方向を変える
- 飽きたらいったん休んで別プロジェクトに行ってもよい

## 9.3. 成功指標 (個人用)

他人に提供するプロダクトではないので、ビジネス指標は持たない。
代わりに **「自分の学びと生活にどれだけ役立っているか」** を主観と客観の両面で追う。

### 9.3.1. メインの指標

**「自分が週に何日開いたか」**
- 週 4 日以上 → 習慣化成功
- 週 2-3 日 → 要てこ入れ
- 週 1 日以下 → 方向転換を検討

### 9.3.2. 学習関連の指標

Insights 画面で自分が確認する指標:

- **Mastery 習得数** — 習得済 concept 数の増加
- **Mastery 成長率** — 週次の mastery 増加量
- **誤概念矯正率** — 繰り返す誤概念が解消されたか
- **正答率推移** — 難易度別の正答率グラフ
- **分野カバレッジ** — 全ドメインに触れられているか (偏り検知)

### 9.3.3. 体感指標 (自己申告)

月 1 で自分に聞く:

- 「業務で前より分かるようになった概念はあるか?」
- 「Tanren で学んだことを実務に活かせたか?」
- 「続けていて楽しいか?」

これらの答えで方向性を調整する。

---

## 9.4. 段階ごとの成功イメージ

| 時点 | 成功の姿 |
|---|---|
| 1 ヶ月後 | 毎日 1 セッション以上解いている |
| 3 ヶ月後 | 50 concept 以上を習得、業務で「あ、Tanren で見たやつだ」体験 |
| 6 ヶ月後 | 150 concept 習得、苦手分野が明確に改善 |
| 1 年後 | 300 concept 習得、エンジニアとして成長した実感 |

## 9.5. 測定方法

### 9.5.1. 自前 DB

- 学習関連 (mastery, attempts) は DB で集計
- `daily_stats` テーブルに日次集計キャッシュ
- Insights 画面で見る

### 9.5.2. アラート

- Sentry でエラー率監視 (開発体験のため)
- Anthropic の予算アラート (コスト管理)

### 9.5.3. ログの活用

- 解いた問題の履歴を定期的に眺める
- 自分が何を学んできたかの足跡として楽しむ

---

## 9.6. 「やめる / 休む」判断基準

個人開発なので「撤退」ではなく「一旦休む」:

- 2-3 週間ログインしていない状態が続く
- 作っていて楽しくない、義務感しかない
- 代替サービスで目的が達成できている
- より学びたい別のプロジェクトが出てきた

こうなったら無理に続けず、学んだことを持って次のプロジェクトへ。
リポジトリは残しておけば将来再開もできる。

---

## 9.7. 方向転換の候補 (気が向いたら)

もし作っているうちに方向性を変えたくなったら:

### D1: 個人用から公開に転換

- 完成度が自分で満足したら、友人や SNS で公開
- フィードバックを受けて改良

### D2: 特定分野特化に絞る

- 「セキュリティだけ」「分散システムだけ」など範囲を絞る
- ニッチだが深い道具になる

### D3: 学習ログとして活用

- 出題よりも「自分が勉強したことの記録」に振る
- Insights を日記的に使う

### D4: 他プロダクトに転用

- 作った仕組み (FSRS スケジューラ、概念ツリー、AI 生成) を別プロジェクトに流用
- Tanren 本体は記念碑として残す

どれも「失敗」ではなく、学びと成果物を次に活かす自然な選択。

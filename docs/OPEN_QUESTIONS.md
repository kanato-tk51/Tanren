# 未決論点 (OPEN QUESTIONS)

## Q1. Streak を長期的にも出さないか

- **現状**: ゲーミフィケーションは「やらない」方針 (`08.9`)。`07` の Home から Streak を削除済み。
- **残る論点**: 半年後に継続率が落ちたら、**義務感を生まない控えめな可視化** (例: Insights の片隅に週別出席ヒートマップ) を足すかどうか。
- **判定**: 3 ヶ月使って自分の継続度を見てから。入れるとしても「at risk 通知」は絶対にやらない。

## Q2. iOS Web Push の実運用可否

- **現状**: MVP では Resend メールを優先、Web Push は Phase 5+。
- **残る論点**: Phase 5 に到達した時点で、iOS PWA の Web Push が本当に信頼できる配信手段になっているか (iOS バージョン, Safari, PWA ホーム画面追加の条件)。
- **判定**: Phase 5 着手時点で実機 (iPhone) でテスト → 1 週間の到達率を見て導入可否を決定。

## Q3. Opus 4.7 を MVP 中に導入するか

- **現状**: MVP は Sonnet 4.6 + Haiku 4.5 のみ。Opus は Phase 2+ (`03.0`)。
- **残る論点**: Sonnet で生成した senior 以上の問題の品質が自分の体感で不足したら、早めに Opus を混ぜる判断をするか。
- **判定**: Phase 1 終了時点 (Phase 1 振り返り) で、生成問題に 👎 を付けた比率を集計。20% 超えなら Opus を senior 以上に限定投入。

## Q4. 「なぜそう答えた?」入力の誘導強度

- **現状**: 任意、10 秒スキップ可 (`07.1.3`)。誤概念抽出は入力があった場合のみ実行 (`03.3.1` の挙動を明示化する必要あり)。
- **残る論点**: 入力なしだと誤概念抽出の精度が落ちる。「誤答時だけは入力を促す UI (1 文だけでも) を強めに出す」か、常に任意のままか。
- **判定**: Phase 3 で Misconception Tracker を本格実装する時に、入力率データを見て調整。

## Q5. 問題の編集フロー

- **現状**: `questions.retired` カラムで「再出題しない」扱いは可能 (`06.2.2`)。
- **残る論点**: LLM が事実誤認した問題を **編集して使い続ける** のか、**常に retired + 新規生成** で済ますのか。前者ならバージョン管理が要るが、後者なら不要。
- **判定**: 当面は後者 (retire + 再生成) で運用し、編集ニーズが 10 回以上出てから UI を作る。

## Q6. Custom Session の NL パース確認 UI の凝り具合

- **現状**: `04.10.1` で「自然言語入力 1 個 (フォーム UI は削る)」、`08.2.1` で「パース確認 UI あり」。
- **合意点**: パース結果を **表示のみ** (読み取り専用カード) し、違和感があれば自然言語で再入力する。初期はフォーム編集は作らない。
- **判定**: 上記で実装 → ドキュメント同期 (`04`, `08` の表現を揃える)。本項目はほぼ結着、実装後にクローズ。

## Q7. 未登録 concept が Custom で指定されたとき

- **現状**: `04.9.3` では動的追加 (Phase 2)。MVP でどう扱うかが曖昧。
- **残る論点**: MVP では (a) エラー、(b) 最も近い既存 concept にフォールバック、(c) 許可して一時的に問題生成だけはする (mastery には反映しない)。
- **判定**: (c) が体験として最も自然。MVP 終盤で実装を足す。

## Q8. Clerk をやめて Lucia/NextAuth に寄せるか

- **現状**: Clerk を採用 (`06.5.1`)。1 ユーザーなら過剰。
- **残る論点**: 公開する気が当面ゼロなら、Lucia/NextAuth で自前セッションの方が依存が減る。Clerk は無料枠内なのでコストではなく「依存の多さ」のトレードオフ。
- **判定**: Phase 0 Day 2 で Clerk 実装してみて、違和感があれば切り替え。今は Clerk のまま。

## Q9. FSRS の自己評価 (1-5) を UI に出すか

- **現状**: `02.4.3` で自己評価併用と記載、`07.1.3` でも入力 UI に触れる。しかし採点結果画面 (`07.2.3`) のモックには出ていない。
- **残る論点**: LLM 採点の結果があれば自己評価は冗長かもしれない。モチベを保つための「自分に厳しく/甘く」調整ボタンとしては意味あり。
- **判定**: Phase 2 (FSRS 統合時) に実装し、1 ヶ月使って入力率が 20% 未満なら削る。

## Q10. テストのカバレッジ目標

- **現状**: `06.8a` で対象は決めたが、カバレッジの数値目標は未定。
- **残る論点**: 個人開発で数値目標を持つ意味は薄いが、FSRS / スケジューラの主要関数は 100% 行カバー目指すべきか。
- **判定**: 数値目標は置かず、「壊れて気付かなかった事故が起きた部分にはテストを足す」帰納的な運用で。
